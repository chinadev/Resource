Swift 中proxy-server.conf启动顺序总结

0人收藏此文章, 我要收藏 发表于9个月前(2012-10-09 13:35) , 已有265次阅读 ，共0个评论

    昨天介绍swift服务启动架构的时候,谈到了proxy启动的时候，根据proxy-server.conf配置文件依次启动服务，下面来看一下我的proxy-server.conf文件

view source
print?
01[DEFAULT]
02bind_port = 8080
03bind_ip = 192.168.4.87
04user = swift
05log_level = DEBUG
06log_facility = LOG_LOCAL3
07 
08[pipeline:main]
09pipeline = healthcheck cache authtoken keystone proxy-server
10 
11[app:proxy-server]
12use = egg:swift#proxy
13all_account_management = true
14account_autocreate = true
15 
16[filter:keystone]
17paste.filter_factory = keystone.middleware.swift_auth:filter_factory
18operator_roles = admin, swiftoperator
19 
20[filter:authtoken]
21paste.filter_factory = keystone.middleware.auth_token:filter_factory
22delay_auth_decision =1
23auth_port = 35357
24auth_host = 192.168.4.87
25auth_protocol = http
26admin_token = zhoubing
27 
28[filter:healthcheck]
29use = egg:swift#healthcheck
30 
31[filter:cache]
32use = egg:swift#memcache
33memcache_servers = 192.168.4.87:11211

其服务启动顺序 会根据 pipeline 逆序启动，但是当请求来的时候，会调用按照顺序调用filter，例如：

pipeline = filter1 filter2 filter2 myapp

这样会先调用 filter1.__call__(env, start_response)然后是filter2._call__(env, start_response)，依次下去最后是myapp.__call__(env,start_response)。 对于Swift来说，主要的流程在于请求来了->auth_token->swift_auth->handle_request，其中每个filter对请求做相应的处理 然后返回。
